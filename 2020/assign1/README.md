# 通俗大白话讲解这份文档
这是一份美国大学CS110课程的编程作业，主题是**凯文·贝肯六度人脉游戏**，分两大编程任务，用C+++底层内存映射技术实现演员人脉最短路径查询工具。

## 一、先讲背景：凯文·贝肯游戏是什么
1. 很早三个大学生发明的好莱坞小游戏：任何两个演员，都能通过「共同出演的电影」搭上线，最多隔6层关系（六度分隔）。
2. 举例子：梅丽尔·斯特里普和杰克·尼科尔森一起演过电影，直接1层；不认识的两个路人演员，中间要串好几个电影、好几个演员。
3. 本次作业就是写一个程序 `./search`，输入两个演员名字，自动输出最短连接链条，超过6层就默认找不到。

## 二、作业整体要求
截止2017年10月4号，晚交不给分；代码混合**底层C裸内存操作** + **高层C++面向对象/STL容器**，既要玩底层二进制文件，又要写广度优先搜索。

整个作业拆成两大核心任务：

# 任务1：实现 imdb 数据库类（最难、底层操作）
程序需要海量演员、电影数据（160万演员、几十万电影），直接普通文件读取太慢，所以不用常规容器存数据，改用**内存映射mmap**把二进制数据文件直接加载进内存，靠裸指针手动读二进制字节。

## 1. 二进制文件存储规则（重点）
有两个二进制文件：`actorFile`（演员数据）、`movieFile`（电影数据），结构一模一样：
1. 文件最开头4字节：存总数量（多少个演员/多少部电影）
2. 紧接着一堆4字节整数「偏移量」：每个数字代表对应演员/电影记录距离文件开头多少字节
3. 偏移数组后面，才是一条条**变长记录**（每个人名字长短、参演电影数量不一样，所以每条记录长度不固定）
4. 好处：偏移数组是有序的，可以用二分查找快速定位目标演员/电影，不用从头到尾遍历。

### （1）单条演员记录二进制格式
一条演员信息是一堆连续字节，固定顺序：
1. 演员名字：C语言字符串，带`\0`结尾；如果名字字节数是奇数，多补一个空字符，保证总字节偶数（内存对齐要求）
2. 2字节短整数：该演员一共演了多少部电影；如果前面总字节不是4的倍数，补两个空字符对齐4字节
3. 一串4字节偏移数字：每个数字指向movieFile里某部电影的位置，代表他参演的影片

### （2）单条电影记录二进制格式
1. 电影名字，带`\0`结尾
2. 1字节年份：只存「年份-1900」，比如1986就存86；如果前面字节总数奇数，补一个空字符对齐
3. 2字节短整数：这部电影有多少演员；按需补空字符4字节对齐
4. 一串4字节偏移数字：每个指向actorFile里一位演员，代表本片全部卡司

## 2. 你要写的两个核心函数
给了你现成类框架，构造、析构函数老师已经写好（负责mmap加载、释放内存），你只实现两个方法：
1. `getCredits(演员名, 电影列表)`：根据演员名字，二分查找二进制文件，读出他所有参演电影，装进C++ vector返回
2. `getCast(电影对象, 演员列表)`：根据电影（片名+年份），读出这部电影全部演员，装进vector返回

### 硬性要求
二分查找必须用C++标准库`lower_bound`，搭配C++11匿名lambda函数做比较；
提供单独测试程序`imdbtest`，可以单独测试数据库读取功能，老师也给了标准答案程序用来对比输出，还要用valgrind检查内存泄漏。

# 任务2：写广度优先搜索BFS，找演员最短人脉链（偏上层C++）
等数据库类写完，再实现搜索功能，对应最终可执行文件`./search`。
## 核心逻辑
1. 输入起始演员、目标演员，用**广度优先搜索**（BFS，保证找到最短路径）遍历人脉；
2. 规则：最多搜6层（路径长度≥7就判定两人无连接）；
3. 工具类/容器使用：
   - `vector`：存电影、演员列表（数据库接口强制用）
   - `list`/queue：做BFS队列，存待遍历的人物路径
   - `set`：记录已经遍历过的演员、电影，防止重复循环卡死
4. 配套自定义`path`类：用来存储、拼接整条演员-电影链路，最后按示例格式打印出来。

## 输出示例
输入两个陌生人，程序逐行打印链路：
A演员和X电影同框B演员，B演员和Y电影同框C演员……直到目标演员。

# 三、代码工具、提交、测试相关说明
1. 代码获取：用hg版本控制工具克隆课程仓库
2. 编译命令：`make imdbtest` 单独测数据库；`make search` 编译完整查询程序；直接`make`全部编译
3. 自测工具：
   - `sanitycheck`：自动对比你的代码和老师标准答案
   - `valgrind`：检查内存泄漏，给了配置文件屏蔽工具自身bug
4. 提交：写完运行课程提供的submit脚本交作业
5. 文件分工：
   - imdb.cc：任务1全部底层代码（你主要写这里）
   - search.cc、path.cc：任务2搜索逻辑代码
   - imdb-utils.h/imdb.h/path.h：头文件，一般不用改
